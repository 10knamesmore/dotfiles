# 深度中文教程编写器

## 适用场景

用户希望为某个**复杂主题**（如数据库内核、Rust 标准库内部、共识算法、操作系统子系统、编译器原理、分布式系统等）产出一套**学习型深度教程**，而**不是 API 文档、不是博客、不是简单 README**。

判断标准（满足任一即应触发）：
- 用户说「写一套教程」「深度讲解」「从零到 X」「沉淀成系列文档」
- 用户希望理解**为什么这样设计**而不是**怎么调用 API**
- 用户预期文档总量数万行、单篇千行级
- 用户希望最终能**自己手写一遍**该系统

## 核心理念

```
教程 ≠ 实现。教程的目标是把读者送到「能自己写」的门口。

✓ 多原理、多对比、多 ASCII 图
✗ 不堆完整可编译实现（顶多伪代码 + 关键片段）

✓ 每章独立子项目（可拆出去单独读）
✗ 不是一个超长 markdown

✓ 每篇必含「动手目标」
✗ 不止于知识介绍

✓ 中文为主、技术术语保留英文
✗ 不要纯英文或纯中文翻译腔
```

## 产出结构

```
docs/<topic>/
├── README.md                       # 顶层索引（章节导航、学习路径、对比表）
│
├── 01-<chapter-slug>/              # 每章 = 一个子目录
│   ├── README.md                   # 章内索引（150-300 行）
│   ├── 01-<sub-slug>.md            # 深度文档 1（≥ 1500 行）
│   ├── 02-<sub-slug>.md            # 深度文档 2（≥ 1500 行）
│   └── 03-<sub-slug>.md            # （视复杂度可有 2-4 篇）
│
├── 02-<chapter-slug>/
│   ├── README.md
│   └── *.md
└── ...
```

简单主题用 2 篇，复杂的（如 HashMap、Channel、async runtime）用 3-4 篇。每章自行决定细分。

## 每篇深度文档的固定结构

```markdown
# 第 NN.M 篇：xxx

## 引言
- 解决什么问题
- 在该章中的位置
- 与上一篇的衔接

## 1. 背景与动机
## 2. 核心原理（多个子章节）
   - 为什么这样设计
   - ASCII 图、对比表
   - 伪代码
## 3. 关键算法 / 数据结构细节
## 4. 实现要点（少量数据结构定义示意，不写完整实现）
## 5. 性能权衡
## 6. 与其他方案对比
## 7. 测试策略
## 8. 核心概念小结
## 9. 动手目标
## 10. 延伸阅读
```

可以追加附录（A/B/C…），常见附录：FAQ、Litmus Test 速记、源码导读、心智模型 cheatsheet、跨语言对比、术语表。

## 章节 README（子目录内）的结构

```
- 本章简介（一段话定位）
- 子篇章列表（表格：编号、标题、主题）
- 学习路径建议（A/B/C 三条）
- 核心概念速览（5-10 条 + ASCII 图）
- 与其他章节的关系
- 动手目标 / 项目检查清单
- 常见误区
- 延伸阅读（书籍、论文、源码）
```

## 顶层 README 的结构

```
- 项目简介
- 章节列表（按部分分组的表格）
- 学习路径建议
- 项目定位（一段话点出"我们在写什么 / 不写什么"）
- 规模数字（章节数 / 篇数 / 总行数）
- 核心概念速览
- 动手清单（完成全部章节后能做什么）
- 相关资源（必读书 / 关键源码 / 经典论文）
- 阅读建议
```

## 硬性要求（强制执行）

1. **每篇深度文档 ≥ 1500 行** —— 低于此视为未完成，必须补足
2. 中文为主，技术术语保留英文（如 `cache hierarchy`、`lock-free`、`happens-before`）
3. **不写完整可编译实现** —— 用伪代码、关键片段、数据结构定义、API 签名，让读者自己动手
4. 大量使用 ASCII 图（内存布局、状态机、时序图、数据流、决策树）
5. 大量使用对比表（与其他方案 / 跨语言 / 性能数字）
6. 每篇末尾的「动手目标」要具体、可验收（不是「理解 X」而是「实现 MyVec，通过 N 个 fuzz 用例」）
7. 每章独立子目录，每章独立 README

## 工作流程

### Phase 1：理解目标（必做）

不要直接动手。先和用户对齐：

```
1. 主题边界是什么？（哪些算入、哪些不算）
2. 目标章节数？（用 AskUserQuestion 给 2-3 个量级选项）
3. 章节细分粒度？（每章 2 篇还是 3-4 篇）
4. 输出位置？（哪个项目目录下）
5. 是否参考已有教程的风格？（指定风格参考文件）
```

特别重要：用户原始的"内容大纲"通常**只是参考**，应当主动建议补充关键的、用户漏掉的知识点。

### Phase 2：制定计划

写一份 plan 文件（plan mode 下到 plan 文件，正常模式下放草稿位置）。plan 必须含：
- Context（为什么要写、写给谁）
- 目录结构
- 每章的内容要点（作为之后 agent prompt 的素材）
- 每章推荐的子文档数量
- 实施方式（一般是「全部并行」）
- 验证方式（行数检查、抽样阅读、风格一致性）

让用户确认后再执行。

### Phase 3：并行 spawn agent（关键）

**每个章节 = 一个独立 agent**（用 `general-purpose` subagent_type）。
全部 agent 在**一个消息内**并行启动，run_in_background = true。

每个 agent 的 prompt 必须包含：
1. 项目背景（已有什么、新增的是第几章）
2. **必读参考**（先 Read 1-3 个已有教程作为风格基准）
3. 输出目录的绝对路径
4. 文件命名规范
5. 完整的「内容要点」清单（不能只是标题，要有 bullet）
6. 每篇深度文档的格式模板（完整粘贴上面的固定结构）
7. 硬性要求（**强调 1500 行**、中英术语、ASCII 图、不写完整实现）
8. 章节 README 应包含的项
9. 完成后用 `wc -l` 汇报各篇行数

agent 数量原则：每章一个 agent。不要尝试一个 agent 写多章。

### Phase 4：跟踪进度

用 TaskCreate 为每个章节建一个任务，agent 跑起来后立即 TaskUpdate 为 `in_progress`。
agent 完成通知到达后，立刻 TaskUpdate 为 `completed`。

### Phase 5：收尾

全部章节 agent 完成后，主对话执行：
1. **用 Read 抽样检查 2-3 篇**，确认风格符合参考样例
2. 写顶层 README（仿已有项目的 README 风格，列出全部章节、学习路径、规模、对比）
3. 更新仓库根的索引（如 `docs/README.md`），把新主题加进去
4. 用 `find` + `wc -l` 输出最终统计

## 风格细节（agent prompt 必须强调）

- 引言要回答"解决什么问题"，不是"本文介绍 X"的废话
- 「核心原理」一节要从**朴素方案不够用**开始推导出更复杂方案
- ASCII 图优先于段落文字解释
- 提到任何性能数字要给出量级（不要"很快"，要"~5x faster"或"P99 < 1us"）
- 章节末尾「衔接到下一章」很重要：让读者明白知识链条
- 不要写"本节我们将学习"这种空话，直接进入主题

## 反模式（agent 容易踩的坑）

| 反模式 | 后果 | 修正 |
|------|------|-----|
| 行数不够就硬填 | 灌水降低质量 | 行数不够说明深度不够，找未覆盖的子主题 |
| 直接给完整 Rust 实现 | 违背"读者自己写"目的 | 改成伪代码 + 关键片段 |
| 缺 ASCII 图 | 文字密度过大 | 关键概念至少一张图 |
| 动手目标太抽象 | 读者不知道做啥 | 写成 checklist：「实现 X，覆盖 N 种边界，跑通 fuzz Y」 |
| 章节 README 重复正文 | 没价值 | README 只做导航 + 速览 + 路径 |
| 只翻译英文文档 | 失去原创价值 | 加入对比、踩坑、性能、个人观点 |
| 一篇文档塞所有内容 | 难以阅读 | 拆成 2-4 篇子文档 |


## 何时不该用这个 skill

- 用户只想要一份 README、API 文档、博客文章 → 直接写
- 用户要的是可运行代码示例 → 用代码 + 简短注释，不需要走教程套路
- 主题太小（一两千字能讲完）→ 不够规模，单文件即可
